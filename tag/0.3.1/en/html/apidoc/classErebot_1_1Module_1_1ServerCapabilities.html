Erebot\Module\ServerCapabilities::getMaxListSize (   $list)

Returns the maximum number of entries a list may contain.

Parameters
opaque$listOne of the LIST_* constants, representing the type of list for which this information must be retrieved.
Return values
mixedReturns the number of entries this type of list may contain or null if no maximum has been defined.
Exceptions
Erebot::InvalidValueExceptionThe value passed to $list does not represent a valid list type.
Note
Even if the server does not specify a maximum number of entries for some list, you should always assume that the server has an implicit value for it, which is usually not that high (like 30 entries or so).
On some servers, several lists are managed together hence the value returned by this method represents the maximum number of entries these shared lists may occupy and not the individual capacity of these lists.

Erebot\Module\ServerCapabilities::hasExtendedNames ( )

Whether extended names are supported by this server. Extended names make it possible to retrieve all modes set for a particular user on a channel using the /NAMES command. This extension is disabled by default and must be explicitly enabled by sending a "PROTOCTL NAMESX" message to the server. When this extension is disabled, the server will only send the highest mode of each user in reply to /NAMES.

Return values
booltrue if the server supports the NAMESX extension, false otherwise.

Erebot\Module\ServerCapabilities::hasHybridConnectNotice ( )

Whether the server supports the Hybrid Connect Notice extension. This extension is aimed at IRC operators and (more likely) IRC services. The Hybrid Connect Notice (HCN) slightly changes the format used by connection notices so that it matches the format expected by the Blitzed Open Proxy Monitor (http://wiki.blitzed.org/BOPM). This includes displaying the user's IP address in connect/exit notices.

Return values
booltrue if the server supports the HCN extension, false otherwise.

Erebot\Module\ServerCapabilities::hasUserHostNames ( )

Whether the server can send full the user!ident@host in a reply to a NAMES command. This makes it possible to build an internal list of addresses without requiring an extra /WHO or /WHOIS command. This extension is disabled by default and must be explicitly enabled by sending a "PROTOCTL UHNAMES" message to the server. When this extension is disabled, the server will only send the user's nickname in reply to /NAMES.

Return values
booltrue if the server supports the UHNAMES extension, false otherwise.
